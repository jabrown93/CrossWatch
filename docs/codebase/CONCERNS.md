### 1) Top Risks (Prioritized)

| Severity | Concern | Evidence | Impact | Suggested action |
|----------|---------|----------|--------|------------------|
| Medium | Webhook auth (Plex HMAC, Emby/Jellyfin shared-secret) fails **open** when `webhook_secret` is unset | providers/webhooks/plextrakt.py:1014-1022 (`if not secret: ... return True`); providers/webhooks/_utils.py:9-12 (`verify_webhook_secret`), consumed identically by embytrakt.py:851-854 and jellyfintrakt.py:1036-1039 | Anyone reachable on the LAN can forge scrobble/rating events with no secret configured; webhook paths are excluded from the `app_auth_gate` middleware (crosswatch.py:464-466). Severity capped at Medium because SECURITY.md already states CrossWatch must not be exposed beyond LAN/VPN — this is a defense-in-depth gap, not an internet-facing exploit | Make `webhook_secret` mandatory (fail closed) when scrobble webhooks are enabled, or surface a persistent UI warning, not just a one-time log line |
| Medium | `cw_platform/url_validation.py`'s SSRF guard is advisory-only: never blocks, never rejects private/RFC-1918 IPs, only flags 2 hardcoded metadata IPs + scheme/path-traversal patterns | cw_platform/url_validation.py:15-45 returns warnings only, no raise; sole call site api/configAPI.py:425-446 only logs at WARN, never blocks the config save | README's "SSRF guarded" framing overstates what's implemented. Server URLs are typically local media servers by design, so blocking private IPs outright would break the primary use case — the gap is real but the threat model (trusted LAN) limits blast radius | Correct the README wording to say "warns only," or add an explicit block for the 2 known metadata IPs at minimum |
| Medium | App-auth is **opt-in, not opt-out** — default install has no username/password, so `app_auth_required()` returns `False` and all `/api/*` traffic is unauthenticated | crosswatch.py:441-499 (`app_auth_gate` middleware) calls `app_auth_required(cfg)` → api/appAuthAPI.py:280-282 (`bool(a.get("enabled")) and credentials_configured(cfg)`) | Fresh/default installs expose the full config/sync/editor/analyzer API surface to anyone reachable on the network with zero auth. Consistent with SECURITY.md's stated LAN-only model, but auth is framed as optional rather than forced | Consider forcing initial credential setup; a partial `setup_lock_required` mechanism exists (api/appAuthAPI.py:319-322) but its covering test is permanently deselected in CI (see Technical Debt) |
| Medium | Sync engine is fully sequential: no concurrency in the snapshot-fetch or pair-execution hot path, despite `ThreadPoolExecutor` being used elsewhere in the codebase | cw_platform/orchestrator/_snapshots.py:588-670 — plain `for name, ops in providers.items(): ops.build_index(...)`, one blocking HTTP call per provider, no pool; cw_platform/orchestrator/facade.py:370 — `for p in pairs:` also sequential. | Low | `docker/run-sync.py` (documented `docker exec` sync trigger) is broken — `Orchestrator()` called with no args against a dataclass requiring `config` | cw_platform/orchestrator/facade.py:50 (`config` field has no default); docker/run-sync.py:24 | Any user following the documented `docker exec python docker/run-sync.py` workflow gets an immediate `TypeError` | Fix the call site or remove the doc reference |

### 4) Performance and Scaling Concerns

| Concern | Evidence | Current symptom | Scaling risk | Suggested improvement |
|---------|----------|------------------|--------------|------------------------|
| Sequential per-provider snapshot fetches | cw_platform/orchestrator/_snapshots.py:588-670 — one blocking HTTP round-trip per provider per feature, no concurrency | Sync duration = sum of all provider fetch latencies, not max | N/A for horizontal scaling (single-instance by design per CLAUDE.md) — this is a single-run latency concern, not a multi-tenant scaling concern | Parallelize with a bounded ThreadPoolExecutor, mirroring the pattern already used in api/probesAPI.py and api/metaAPI.py |
| Sequential per-pair execution | cw_platform/orchestrator/facade.py:370 — `for p in pairs:` | Multiple configured sync pairs run one after another | Same as above — latency, not scale | Consider pair-level parallelism where pairs don't share provider rate-limit budgets |
| Unbounded, never-cleared module-level caches in Trakt history resolution | providers/sync/trakt/_history.py:871-873 (`_SHOW_PATH_CACHE`, `_SEASON_EP_CACHE`, `_EP_RESOLVE_CACHE`) — none has a `.clear()` call anywhere, unlike the equivalent Emby (`_item_meta_cache.clear()`, providers/sync/emby/_history.py:589) and Jellyfin (`_SERIES_META_CACHE.clear()`, providers/sync/jellyfin/_history.py:395) caches | Process memory grows monotonically with distinct shows/seasons/episodes ever resolved across the process lifetime | Bounded in practice by library size, but a long-running container with large/rotating libraries has no eviction | Add the same `.clear()` hook used by the Emby/Jellyfin equivalents, or switch to an LRU-bounded cache |
| In-memory caches assume single-instance deployment | CLAUDE.md states the app is designed as single-user/single-instance | N/A | Multi-instance deployment would break these caches' correctness — but this is explicitly out of the documented deployment model | No action needed unless the single-instance assumption changes |
